流淌着的时光就像镜子里的阳光，那么清晰，那么虚幻。
母亲曾经说我是可以控制镜子的人，所有的东西在我的思绪里来则应、去则散，永远不会留下任何的影子。因为我总在笑吗？我并不十分的清楚。总有人问：你快乐吗？我总在笑啊！为什么她们还是认为我不快乐？我并不伤心啊！人真的是很奇怪的生物。就像母亲，很害怕我得自闭症，却又后悔带我进入别人的圈子。似乎什么东西都可以在人的思维里共存，可是唯一不可以共存的却也是人。

经常梦到同一个人。看不清她的样子。她身边似乎有很多的人。可是，每次看到她自己总是一副冷冰冰的样子。她身边的人似乎都很怕她，也很迁就她，除了我。一开始很高兴有人可以陪着自己，渐渐的却不再想梦见她。直到那天胃痛得昏过去，清醒之后才明白，之所以会高兴只是因为自己太寂寞。而自己本身就是梦，就是幻像，更何论梦中的梦。终于看清了她的样子，她说她找了我很久很久了。我只是笑。她根本就找不到我的。她是我梦中的幻影，我是现实中的梦。　

母亲说：“如果你真的在现实中碰见她呢？”我说那是不可能的。那只是我的梦，即使她真的出现也不会认识我的。我没说的是，她的生活环境是我所不喜欢的。就如我在梦境中看到的，她跟我是活在两个世界的人。我笑着对母亲说：“也许是她前世欠我的吧！”可是，我们都很清楚：人是没有前世的。我们也不需要前世。记忆是会被记录下来的，可是记忆的速度快于记录的速度，活在差距里的我们如果执意于某个记忆，即使记忆残留，剩下的也只会是死亡吧！看着外面下着的大雨，突然觉得也许我可以不再寂寞，即使我真的一个人活在世界的某个角落里。　

时间越久，看清的东西越多。不论是梦境中的还是现实的，一切都那么的清晰。那些清晰的东西就像纷飞的雪花，飘落无声。当太阳出来的时候，所有的都消散无痕。所以，看到“滕六掌雪，青女弄霜”时才明白自己有多天真。只是再怎么的天真都是会在时光里湮灭成灰的，有的甚至连灰烬都不存在。唯一留下的只是那些被记录下来的记忆。就像当初认识雨的自己，在经过时光的洗礼下，由无知变得彻底的漠然。也明白，原来世界上之所以会有所谓的注定是因为人的偏执。　
　　
似乎要结束了，我对母亲这样说道。她说知道。再回首，细想过往的一切，对我们来说都只是梦吧！有时候在想，以我的一切换来的东西是否真的是我所希望的存在。她们会如何？连我自己都不清楚。站在别人的世界外，不去期待，不去进入，不投入感情，当有人离开的时候也就不那么的伤感了。曾经以为会这么想只是为了不再受伤，直到现在才明白，其实这些与不再怨恨是一样的道理。之所以不再怨恨是因为早已经无力再去爱了。没有爱的人就像没有影子一样，是不能活在阳光之下的。更不可能进入人的世界 。

姝君：

不知为何，明明想和你说话，却骗你说，阳光明媚，风雨正好。不必嘲讽我，即使你笑出声来，我也当是天籁。更不必怀有敌意，你所有的心计，我都当作是你对我的心意。

我的宿命分为两段，未遇见你时，和遇见你以后。

人世间有太多的嘈杂，未遇见你时，曾爱过一个人，恨过一个人，在乎过一个人，鄙视过一个人，为了一个人努力付出过，也有过那么一个人为我付出，但那些人却从来不是对的人，或许这些经历，都是在磨砺我。转眼间这些都变成了过去，我怀着对感情的勿滥做着真实的自己，心口相一，仰望星空，广袤无垠，好多年过去了，直到我遇见你，耳畔响起一个声音：“就是她”，没有理由，没有原因，更没有方法去判断，就是知道。

遇见你以后，期盼每天都能见到你，计划着陪你去那些能给你惊喜美食的餐厅，也想抱着你一起看你喜欢看的电视节目，或者静静地看着你做着你喜欢做的事情。

从此，从来没有看书习惯的我，开始每天看书，也爱上了看书。因为工作放下的健身，也重新捡起，每天坚持早起去锻炼，再苦再累也不怕，因为想做一个更好的自己，展现给你。

从此，心中也有了一个挂念的人，她的一哭一笑，她的一言一行，都成了我的晴雨表。情绪的波动，也是因为你。8月28号那天，你告诉我，你决定回家了，心中的美好全部崩塌，即使你不告诉我，我也知道，你或许喜欢我，但并不会为了我留下来，即使，我花尽全身勇气跟你说：“我养你”。

在一起20天，短暂的爱情却胜过纠缠一辈子。至少在七夕节的那一刻，我们的心在一起。

你治好了我的忧郁，而后赐我悲伤。忧郁和悲伤之间的片刻欢喜，透支了我生命全部的热情储蓄。

好想去饮一些酒，让我的灵魂失重，让风吹走。

姝君，我终究将是你的路人，不如从此刻开始，我变成是你的路人，就像我们从未相识，即使会让我感觉变成了全世界的路人。把那些美好都留在记忆里，以后回想起你，我的丑脸也能不由自主的微笑。

愿从此，你能快快乐乐，找到一个你觉得对的人，带回家，告诉你的父母，我非他不嫁。

愿你能嫁给爱情，而不是因为合适！

风虽大，却绕过了我的灵魂。

不知不觉在公司做接口测试已经接近3个月了。由于之前一直在做自动化平台的开发，所以上手时走了不少弯路，而且老实说现在还在走弯路中，所以今天就总结下目前做接口测试的一些感悟。

##什么是接口测试？
先搬运一下维基百科上的英文解释：

API testing is a type of software testing that involves testing application programming interfaces (APIs) directly and as part of integration testing to determine if they meet expectations for functionality, reliability, performance, and security.[1] Since APIs lack a GUI, API testing is performed at the message layer.[2] API testing is now considered critical for automating testing because APIs now serve as the primary interface to application logic and because GUI tests are difficult to maintain with the short release cycles and frequent changes commonly used with Agile software development and DevOps).[3][4]

翻译过来就是：

API 测试是一种作为集成测试的一部分，通过直接控制被测应用的接口（API）来确定是否在功能、可靠性、性能和安全方面达到预期的软件测试活动。[1]由于 API 都没有 GUI 界面，API 测试都是在通讯层进行的。[2]现在 API 测试在自动化测试中有着很重要的地位，因为 API 一般是应用逻辑的主要接口，同时 GUI 测试在敏捷开发和 DevOps 的快速迭代和频繁变更中很难维护。[3][4]

目前我们项目组都是前后端并行开发，并没有先后顺序，因此，我认为接口测试开始的具体时间点应该是在后端开发与前端约定好API定义，并提供出接口文档后就开始。而目前项目组中，前端开发并没有搭建mock server为其提供mock服务，主要依赖与后端约定好联调时间进行自测，因此联调时因为后端接口自测不足，因而导致提测质量很低，接口一堆bug。如果能够在早期就把接口层的问题暴露出来，后续前后端联调时也能提升效率和提测质量，测试也不用看着这一堆的bug而头疼。

##接口测试的目的
我觉得目的就两个：

  1. 在开发接口联调前，尽早暴露出接口的问题，减少前后端联调时间以及前端返工工作量，提高提测质量

  2. 接口测试执行快，反馈快，可以当成单元测试一样测，测试过程中还可以加入到CI中定期执行，可以减少回归量，持续比对接口返回结果，保证代码质量，提高效率。

    那为什么说要像测单元测试一样去覆盖接口测试，因为越是底层的东西覆盖率更应该提高，当成单元测试一样测，出现问题可以迅速找到根源，也不会因为一个bug block住之后的流程。

##我的感悟
我刚开始时对于写接口测试的主要思路是结合业务去设计用例，如query_game_charge_record_list查询游戏充值记录列表，如有翻页参数，那就试试默认值、有效值、最后一页、最后一页+1、无效值。针对各个参数和业务场景去设计用例。但是，其实按照这个设计思路去做，我设计的用例是有问题的。

举例说明，假设我要验证双因子登录校验接口，工作流如下：

  1. 登录app，服务器返回token
  2. 激活设备，服务器返回双因子校验信息
  3. 发送密保验证码
  4. 双因子之手机短信登录校验，包含双因子校验信息校验Type(checkType)，校验内容(checkContent)，是否免输入将军令/密保卡(isEnterAssistPwd)，激活类型(bizType)等主要字段。服务器会返回是否成功，激活账号以及appSessionId

按照开始时的设计思路去设计的验证双因子登录校验的用例如下：

  1. 用登录接口获取token，并验证token是否返回，以及返回结果是否正确
  2. 调用设备激活接口，获取该账号的双因子校验信息，其中token来自第一步的返回结果
  3. 调用发送密保短信接口，并获取验证码
  4. 调用双因子校验接口，其中校验信息校验Type，校验内容，是否免输入将军令/密保卡，激活类型来自第2，3步的返回值，查看返回账号是否为登录账号 5.调用查询游戏充值列表接口，查看游戏列表是否正常返回，其中appSessionId是双因子校验接口通过后的返回值，只有带该参数，app接口才能调通。

这样看来似乎没有什么问题，但其实这个用例本身存在一个很严重的问题：接口成为了手段，验证点其实是功能。

通过查资料以及逛论坛，我了解到关于接口测试用例，主要分为两类：

  1. 单一接口测试，调用一个接口就是一个用例，而且每个用例只测一个点 2. 多接口的测试用例，会调用多个接口，并且各个接口直接存在参数传递

接口测试用例要简单，并且每个接口都应该至少设计一个用例是使用默认参数调用单个接口的。使用单功能单用例的覆盖，目的就像前面说的，是为了更方便准确的发现问题，并快速定位，而且也不会因为某一个bug block住后面的流程。而像我上面设计的用例，已经不是用来验证双因子之密保手机登录校验了，而是验证通过密保手机类型来验证双因子登录成功的功能了。

那么发朋友圈的接口应该有什么用例呢？我按照现在的测试接口的思路重新想了一下，主要有这几个：

  1. 有checkType，checkContent，isEnterAssistPwd，bizType，预期返回校验成功
  2. 有checkType，checkContent，isEnterAssistPwd，无bizType，预期返回校验失败，报错：“由于您长时间未使用，登录状态失效，请重新登录【060071】”
  3. 有checkType，checkContent，bizType，无isEnterAssistPwd，预期返回校验成功
  4. 有checkType，isEnterAssistPwd，bizType，无checkContent，预期返回校验失败
  5. 有checkContent，isEnterAssistPwd，bizType，无checkType，预期返回校验失败
  6. 有checkType，checkContent，isEnterAssistPwd，bizType，预期返回校验成功，同时返回的appSessionId，有效且传参到别的接口也能正确返回值等等

当然还有非法参数，空值等的测试，这里我就不全部列出来了。其实每一个用例测试一个点，然后发布成功这个返回值是否正确由一个单独的用例来覆盖就足够了。但是这种按着业务功能来设计的接口用例，从某一方面讲，是能够确保业务场景的功能以及流程是否畅通，但是却不能完全替代前端的重复性功能测试，所以覆盖一种足矣。

接口测试的本质是通过测试参数的排列组合验证返回值/数据库变更是否符合预期，从而确定接口相关代码是否正确。从业务角度考虑的意思是这些排列组合不是随便想出来，而是业务中有可能出现的排列组合。

所以做好接口测试并不像之前想象得那么简单，但也没有刚开始的时候做得那么艰难。不管怎样，既然开始了，那就要想办法把它做好。接下来我会使用新的设计用例思路做剩下的接口测试用例。感悟中如果有不对的地方，欢迎大家及时提出。
